### 觀念 1：什麼是「後端路由」？為什麼要「多開一條路由」？

**路由（route）** 就是一個網址對應的處理程式。在 Next.js 裡，`src/app/api/xxx/route.ts` 這個檔案路徑，就會自動對應到 `你的網站/api/xxx` 這個網址。你打開瀏覽器連到 `/api/recommend`，Next.js 就會執行 `route.ts` 裡寫的程式碼，回傳結果。

今天新增了兩條路由：
- `src/app/api/recommend/route.ts`：處理「幫我推薦餐廳」這個核心功能。
- `src/app/api/photo/route.ts`：處理「拿餐廳照片」這件事。

**為什麼照片要「多開一條路由」，不能直接讓前端去跟 Google 要圖？**

Google 的照片網址長這樣：`https://places.googleapis.com/v1/{照片編號}/media?key=你的API金鑰`。如果前端的 `<img src="...">` 直接寫這個網址，代表你的 API 金鑰會整個出現在網頁的原始碼裡——任何人打開瀏覽器「檢查元素」都看得到，等於把金鑰公開給全世界，別人可以拿去濫用、產生你的帳單。

解法是讓**後端**（只有伺服器自己看得到、使用者看不到程式碼）去跟 Google 要圖，再把圖片內容原封不動轉交給前端。前端只知道「跟我自己的網站要 `/api/photo?name=xxx`」，完全不知道 Google 金鑰的存在。這個做法有個常見名稱叫 **proxy（代理）**：後端當中間人，擋在使用者跟真正的外部服務之間。

這也是為什麼一開始申請 API 金鑰時，我特別提醒「金鑰只能放後端」——今天的照片功能就是這個原則的具體案例。

### 觀念 2：為什麼要讓 LLM 輸出「結構化 JSON」，而不是直接寫一整段話？

一開始的做法：請 Gemini 直接生成一整段推薦文字（開場白 + 3 段介紹全部黏在一起）。這樣做的問題是：如果我想把「開場白」跟「每間餐廳的介紹」分開放在畫面不同位置（例如你今天要求的：開場白獨立、每段介紹放進對應的餐廳卡片），程式沒辦法知道這一大段文字裡，哪一句是開場白、哪一段對應哪一間餐廳——文字對電腦來說只是一串字元，沒有「結構」。

解法是請 LLM 不要輸出「一段話」，而是輸出「一份資料」，明確規定格式：
```json
{
  "opening": "一句同理心的話",
  "reasons": [
    { "placeId": "餐廳的Google編號", "reason": "這間餐廳的推薦理由" }
  ]
}
```
程式收到這份資料後，就可以直接把 `opening` 顯示在一個位置、把每個 `reason` 依照 `placeId` 對應塞進正確的餐廳卡片。這種做法叫 **structured output（結構化輸出）**，現代的 LLM API（Gemini、Claude、GPT 都有）都支援：你可以告訴它「請照這個格式回答」，它就會盡量產生符合格式的 JSON，而不是自由發揮的文章。

這個功能原本寫在 `src/lib/gemini.ts`，後來整支檔案換成 `src/lib/claude.ts`（原因見下方「觀念 5」），但「結構化輸出」這個做法本身沒變，只是換一家 LLM 供應商的語法。

### 觀念 3：深色模式讓文字「隱形」的 bug

你回報「年齡/性別選單」「定位按鈕」「餐廳卡片」的文字看不到，背景跟文字都是白色。

原因：`globals.css` 裡原本有一段「跟隨系統深色模式」的設定——如果你的手機/電腦是深色模式，網頁文字顏色會自動變成接近白色。但頁面背景我是用 Tailwind 寫死成淺色的，沒有跟著深色模式一起切換。這幾個地方剛好沒有另外指定文字顏色，就變成「淺色背景 + 自動變白的文字」，等於文字消失。

解法：拿掉「跟隨系統深色模式」這段設定，讓網頁固定使用淺色主題；並加上 `color-scheme: light`，這會告訴瀏覽器「這個網頁是淺色的」，連瀏覽器自己內建的表單元件（像下拉選單）樣式也會強制用淺色，不會被系統深色模式影響。

這是一個很典型的「CSS 繼承」問題：文字顏色（`color`）預設會從外層容器一路往內傳，如果內層元素沒有自己指定顏色，就會用外層傳下來的值，而外層的值又受系統設定影響，中間沒有明確鎖死，就容易出現這種在特定環境下才會發生的 bug。

### 觀念 4：首輪/次輪從「分頁切換」改成「同頁捲動」

原本的做法是做兩個分頁按鈕（「首輪推薦」「次輪推薦」），點擊哪個就顯示哪組資料，同一時間只看得到一組。

你的回饋是：手機使用者習慣「往下滑」勝過「點分頁切換」。所以改成：兩組資料**同時顯示在同一個頁面**，次輪接在首輪下方；當次輪資料載入完成，程式會自動把畫面平滑捲動到次輪區塊的開頭，讓使用者不用自己找。

技術上用了 React 的 `useRef`（取得某個畫面元素的參照）搭配瀏覽器原生的 `scrollIntoView({ behavior: "smooth" })`（捲動到該元素、且是平滑捲動而非跳動）。程式碼在 `src/app/components/ResultsView.tsx`。

### 今天用到的其他小知識
- **Tier fallback（分層降級搜尋）**：心情關鍵字分 3 層，越精準的先試，找不到 3 家才降級用更廣泛的詞，避免搜尋結果是空的。這在 `src/lib/recommend.ts` 的 `runTieredSearch` 實作。
- **硬性 vs 軟性條件**：飲食限制（素食、過敏）是硬性，不符合直接從候選清單剔除；健康方向、辣度是軟性，只影響 LLM 怎麼措辭關鍵字，不會真的把餐廳踢掉。
- **為什麼 Gemini 模型名稱要用 `gemini-flash-latest` 而不是 `gemini-2.0-flash`**：實測時發現 `gemini-2.0-flash` 的免費額度已經是 0（Google 常常會逐步淘汰舊版模型的免費資源），改用官方提供的「別名」，它會自動指向目前建議使用的最新版本，以後 Google 換模型也不用回來改程式碼。

### 觀念 5：為什麼同一個功能，一次「推薦」動作原本要呼叫 LLM 到 7 次？

推薦邏輯有兩層會呼叫 LLM：
1. 心情關鍵字分 3 層（Tier1/2/3），原本**每一層都呼叫一次 LLM** 去微調搜尋字串。
2. 如果候選餐廳不夠 3 家，會觸發「次輪放寬」（距離 +1km、預算 +1 級），這時又把上面 3 層**整套重跑一次**。

3 層 × 2 輪 = 最多 6 次微調呼叫，加上最後生成推薦文案 1 次，一次「推薦」動作最壞情況要打 7 次 API。

後來發現一個關鍵：**「次輪放寬」只改變距離跟預算，完全沒有改變心情、動機、會員資料**——而搜尋字串微調這個函式，輸入只跟心情/動機/會員資料有關，跟距離/預算完全無關。也就是說，次輪放寬時重新呼叫 LLM 去微調搜尋字串，得到的結果理論上跟首輪一模一樣，等於是白白多打一次 API、多花錢又浪費時間。

解法：把「算搜尋字串」這件事從「查候選餐廳」這件事裡拆出來，**只算一次，兩輪都重複使用**。同時 Tier2/Tier3（本來就是「精準度較低的保底選項」）也不用再麻煩 LLM 客製化，直接用字典裡的關鍵字硬串起來就好，個人化程度差一點點沒關係，換來的是呼叫次數大減。改完之後，最壞情況從 7 次降到 2 次。

這是一個很典型的「重複運算」問題：程式常常會不小心把「同樣的東西，因為寫在迴圈或流程的不同分支裡」而算了好幾遍，只要把「輸入沒變，就不用重算」的部分抽出來，效能跟成本就能一次砍掉一大截。程式碼在 `src/lib/recommend.ts` 的 `buildTierQueries` 函式（負責「只算一次」）跟 `runTieredSearch` 函式（負責「拿現成的字串去查」）。

### 觀念 6：為什麼「Google 帳號有試用額度」不代表 Gemini API 就免費？

這是今天踩到、也最值得記住的一個坑。

Google Cloud 主控台顯示你有一大筆試用額度（例如 $9,422 美元，到期日還很久），直覺會覺得「這個額度應該可以隨便用」。但實測發現 Gemini API 的計費，走的是另一套叫「Cloud Prepay」的獨立系統，**不會自動扣抵你在 Google Cloud 主控台看到的那筆試用額度**，而是要你在 AI Studio 那邊另外用信用卡儲值（最低 NT$1,000），而且這筆錢**不可退款**、只能用在 Gemini API 上。

換句話說：同一個 Google 帳號底下，不同的 API 產品可能各自有各自的計費規則，「帳號有額度」不等於「這個 API 免費」，一定要去該 API 專屬的帳單頁面確認一次，不能只看主控台首頁那個總覽數字。

這也是為什麼最後決定不儲值 Gemini（不想花一筆不可退的固定金額），改回申請 Anthropic 的 Claude API——雖然 Claude 一樣要付費，但是「用多少扣多少」的隨用隨付模式，不用先囤一筆錢進去。

### 觀念 7：換 LLM 供應商，為什麼只改一個檔案就好？

從 Gemini 換成 Claude，整個專案只動了 `src/lib/gemini.ts`（刪除）→ `src/lib/claude.ts`（新增），還有兩個檔案裡各改一行 import 路徑，其他推薦邏輯、頁面、UI 元件完全沒動。

能做到這樣，是因為一開始設計的時候，就把「呼叫外部 LLM」這件事包成兩個固定的函式：`refineSearchQuery`（微調搜尋字串）跟 `generateReply`（生成推薦文案），其他程式只知道「呼叫這兩個函式會拿到什麼」，不知道、也不需要知道背後究竟是 Gemini 還是 Claude 在處理。這種「把會變動的細節包起來，只露出固定的介面給外部用」的做法，讓「換供應商」這種原本聽起來很麻煩的事，變成只需要重寫一個檔案的內部實作。

兩家的語法細節不一樣，值得記一下：
- Gemini：用原始的 REST API（`fetch` 打 `generativelanguage.googleapis.com`），結構化輸出要自己組 `responseSchema`。
- Claude：用官方的 `@anthropic-ai/sdk` 套件，結構化輸出改用 `zod`（一套定義資料格式的工具）搭配 SDK 內建的 `zodOutputFormat`，寫法更簡潔，也讓 TypeScript 能直接檢查回傳資料的型別對不對。

### 觀念 9：為什麼「限制搜尋範圍」的參數，結果還是會超出範圍？
Google Places Text Search 的 `locationBias`（我們拿來限制搜尋範圍用的參數）從名字就看得出來，它只是「偏好」（bias），不是「限制」（restriction）。意思是 Google 會**傾向**在這個範圍內找結果，但如果它覺得範圍外有更符合關鍵字的地點，還是可能回傳。這跟直覺「我設定 2 公里搜尋半徑，結果就該全部落在 2 公里內」不一樣。

解法不是去改 Google 那邊的參數（因為 Google 就是這樣設計的），而是**在自己的程式裡再做一層硬性把關**：不管 Google 回傳什麼，程式自己算出實際距離，超過上限的直接丟掉。這是一個常見的原則：**外部服務給的「篩選」通常只是最佳化建議，真正要保證的規則，要在自己的程式裡再檢查一次。**

### 觀念 8：金鑰「申請成功」不代表「能用」
今天學到一個細節：API 金鑰的「建立」跟「能不能扣款使用」是兩件分開的事。Anthropic 允許你先建立金鑰，但真正呼叫 API 時才會檢查帳戶餘額夠不夠——所以「金鑰格式正確、認證通過」（回應是 400 而不是 401）跟「帳戶有錢可以用」是兩個獨立的檢查點，除錯時要分開看錯誤訊息才知道卡在哪一關。

這次因為 Anthropic 的儲值卡關問題還沒解決（且已經卡了一段時間），與其一直等，改用 OpenAI 的 `gpt-4o-mini` 讓功能先能動起來、驗證整體流程沒問題。之後如果 Anthropic 那邊儲值成功了，因為介面設計沒變，隨時可以把 `src/lib/openai.ts` 換回 `src/lib/claude.ts`（程式碼還留著 git 歷史紀錄裡）。

### 觀念 13：半透明背景圖為什麼要「另外包一層白底」，不能直接把圖片本身調透明就好？
如果直接把 `<img>` 的 CSS `opacity` 調低，效果沒問題（圖片整體變淡），但圖片本身沒有「文字保護」的功能——不管圖片多淡，只要文字直接寫在這張圖片上面，兩者的顏色還是有機率疊在一起變得難以分辨（例如圖片裡剛好有一塊跟文字顏色相近的區域）。解法是在「淡化的背景圖」跟「文字內容」之間，再加一層**半透明的白色卡片**（`bg-white/80`，80% 不透明的白色）。這一層卡片本身也會透一點點背景圖的顏色出來（維持「看得到背景」的視覺效果），但因為底色夠白、夠不透明，文字疊在卡片上永遠是「深色字＋淺色底」，不會受背景圖內容影響。這是「用一層穩定的中介層，而不是逐一調整每個文字顏色」的做法，好處是不管背景圖換成什麼內容，文字對比度都有保障，不用每次換圖都重新檢查。

### 觀念 10：為什麼「安全提醒」要用彈窗，不能只靠頁面角落一行字？
過敏原過濾本質上是「盡力而為、不保證安全」的功能——關鍵字比對不可能涵蓋所有真實料理成分。原本只在表單裡加一行紅字，问题是**使用者不一定會往下滑或注意到**，尤其手機畫面小、視線容易只停在剛剛按下去的按鈕上。彈窗（modal）會強制蓋住整個畫面、擋住後續操作，直到使用者按下確認才會消失，等於是用互動流程本身保證「這句話至少被看過一次」，而不是賭使用者眼睛會掃到那行小字。這是「重要但使用者可能忽略的資訊」跟「一般性說明文字」該用不同呈現方式處理的例子。

### 觀念 11：「自動幫使用者做決定」跟「讓使用者自己選」，哪個比較好？
原本首輪的邏輯是：查不到3家就自動放寬距離/價位、自動降級關鍵字，湊滿3家才給使用者看，使用者完全不知道背後發生過幾次「降級」。這樣做的問題是：使用者以為自己拿到的是「精準」推薦，但實際上可能已經是好幾層放寬後的結果，過程對使用者是隱形的。

改法是把「要不要放寬」這個決定的主控權還給使用者：首輪誠實地只用精準關鍵字查，查到幾家就是幾家（包括可能是0家），只有真的0家時才在背景做最基本的救援（避免畫面完全空白），然後明確問使用者「選擇不多，要放寬搜尋嗎？」，由使用者自己按按鈕決定，而不是系統默默幫他放寬。這是把「系統自動優化」跟「讓使用者知情並選擇」這兩種設計哲學做取捨的例子——不是哪個絕對比較好，而是看這個放寬動作會不會改變使用者原本的期待（這裡因為使用者是設定好心情才查詢的，放寬关键字/距離确实会改變推薦的精準度，所以选择讓使用者知情同意，比系統自作主張更合適）。

### 觀念 12：三個很像的「找餐廳」動作，為什麼可以共用同一套程式碼？
這次新增了三個情境：首輪、換一批（次輪）、放寬搜尋，聽起來是三個不同功能，但拆開來看，「換一批」跟「放寬搜尋」的搜尋邏輯其實一模一樣（都是 Tier1→2→3 查到湊滿需要的筆數，不夠再放寬距離/價位），唯一的差別只有「要湊到幾家」跟「要排除哪些已經看過的餐廳」這兩個參數。所以程式碼把這套邏輯寫成一個共用函式 `runFullRelaxCascade`，讓「換一批」跟「放寬搜尋」都呼叫同一份，只是傳進去的參數不同。如果照字面上「三個功能」的直覺去寫，很容易變成複製貼上三份幾乎一樣的搜尋迴圈，之後改一個地方（例如放寬幅度要調整）就要記得同步改三個地方，容易漏改、也容易讓三份邏輯漸漸長歪。判斷「這是不是同一件事」不能只看功能的名字，要看它們背後實際在做的運算是不是真的一樣。

## 2026-07-03（續2：距離機制討論與修正）

### 起因
複查距離放寬機制（「保底救援」「換一批」「放寬搜尋」都會把搜尋半徑 +1km 再查一次）時，發現硬性距離上限（`computeMaxDistanceKm`，以使用者原始輸入距離為準，不隨放寬動作放大）可能讓「+1km」這個放寬動作在短距離情境下形同虛設——例如輸入0.5km時上限只到0.6km，半徑擴大到1.5km也沒用，能新增的合法範圍只有0.1km。討論後確認這是刻意設計（距離上限代表使用者選擇距離時隱含的交通工具可達範圍，不該因為「放寬」或「使用者主動選擇進入次輪」就跟著放大——這兩者是「預期管理」跟「物理可達性」兩個不同的問題，後者不會因為使用者主動點按鈕就改變）。

### 用真實 API 呼叫（非 mock）測試距離偏誤的根本原因
直接呼叫 Google Places `places:searchText`，比較「原始半徑」與「+1km放寬半徑」的查詢結果，依實際 haversine 距離分區間統計，測了：
1. 現行多關鍵字組合查詢字串（tier1 LLM 輸出、tier2/tier3 拼接格式）：8組測試（2地點×2距離×2字串），「原始範圍內」恆為0，超過放寬半徑仍被回傳的比例高達 17~21/20~21 筆。
2. 單一短關鍵字（「便當店」「餐廳」等）：都市（中興大學，密度高）改善有限；郊區（中正大學，密度低）改善明顯，2km時合法範圍內可達9~14/20~21筆。
3. 追加 `rankPreference: "DISTANCE"`（Google Places Text Search 支援的排序參數，查文件確認存在且可用於分類查詢）：郊區2km「餐廳」從「2筆合法+19筆超出」變成「20筆全部落在放寬半徑內」，改善非常明顯；都市與極短距離（0.5km）幫助有限，這部分看起來是 Places API 對極小圓形 bias 的結構性限制，不是排序參數能解決的。

結論：**locationBias 對現行多關鍵字組合查詢字串幾乎完全失效，不是「換一批」「放寬搜尋」的+1km幅度不夠大**——真正該優化的是查詢字串怎麼組、排序參數有沒有用對，不是搜尋半徑本身。這跟原本「以為是保底/放寬機制的距離增量沒調好」的假設是不同的問題根源。

### 決定與修改
1. `src/lib/places.ts` 的 `searchPlacesText`：request body 加上 `rankPreference: "DISTANCE"`（低成本、有實測數據支持、無明顯副作用）。
2. `src/app/components/QueryForm.tsx`：距離滑桿下限從 0.5km 改成 1km（原因：使用者選 500m 這種極短距離時，locationBias 幾乎必定失效——不是「附近真的沒有店」，而是 API 排序機制的結構性限制，讓使用者以為「500m內應該有很多選擇」但系統做不到，容易造成錯誤預期；且現實中密集餐飲聚集通常需要辦公商圈或學區這種特殊密度，不是一般情況）。

### 觀念 14：同一個技術限制，可以用「使用者體驗」跟「工程」兩種角度處理，該選哪個？
發現「500m內查不到店」有兩層原因：(1) 附近真的沒那麲多餐廳；(2) Google Places 對極小搜尋半徑的地理偏誤本來就弱，不管關鍵字或排序參數怎麼調都救不了。這次沒有選擇「繼續在工程面想辦法救500m情境」（例如換用 Nearby Search API、改用矩形 `locationRestriction`），而是選擇「產品面直接不提供500m這個選項」（滑桿下限拉到1km）。判斷依據：如果一個技術限制沒辦法在合理成本內修好，與其讓使用者看到功能「看起來能選但實際上做不到」，不如在選項層級就不提供，這樣使用者的預期從一開始就是準的，不需要靠額外文案去解釋「為什麼你選了卻找不到」。
