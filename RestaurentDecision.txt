「用餐選擇小幫手」AI Agent 實作計畫：

階段一：Google Maps Platform 基礎環境與 API 設定
AI Agent 需要合法的資料來源來獲取真實餐廳資訊，必須先完成以下設定：


申請專案與啟用帳單： 登入 Google Cloud Console 建立新專案（例如：Restaurant-Recommender），並綁定信用卡啟用帳單功能，以確保享有每月 200 美元的免費額度 。


啟用核心服務： 於「API 與服務」中啟用 Places API，這將是 AI 進行文字搜尋、附近搜尋與獲取餐廳詳細資訊（評價、營業時間）的核心 。若有前端顯示地圖需求，可視情況啟用 Maps JavaScript API 或手機端的 SDK 。


安全與金鑰管理： 建立 API 金鑰後，務必設定「應用程式限制」（如限制特定網域或 IP）與「API 限制」（僅允許 Places API 使用），以防止被盜用產生高額帳單 。

階段二：定義 AI 核心推薦邏輯（參數轉換字典）
在呼叫 API 之前，AI Agent 需要具備一套轉換邏輯，將使用者的主觀感受轉化為具體的搜尋條件。

1. 心情參數（情緒性飲食對應）

壓力大、焦慮： 轉換為「酥脆炸物」、「重口味」、「帶有辣度的料理」 。


悲傷、低落： 轉換為「甜點」、「熱湯」、「濃郁滑順澱粉類」 。


開心、想慶祝： 轉換為「大份量」、「適合多人分享」、「精緻的高級餐點」 。


疲勞、需要專注： 轉換為「清爽」、「高蛋白質」、「富含維生素的原型食物」 。


生氣、煩躁： 轉換為「帶骨肉類」、「冰涼降溫的食物」、「極度酸辣的口味」 。

2. 動機參數（情境與預算設定）

簡單吃： 增加篩選條件：快餐、小吃、定食（出餐速度快、翻桌率高） 。


聚會聚餐： 增加篩選條件：空間寬敞、提供大桌的餐廳 。


犒賞自己： 增加篩選條件：裝潢精緻、食材等級較高的高級餐廳 。

階段三：系統提示詞（System Prompt）模板建置
將以下 Prompt 模板內建於 AI Agent 核心，確保其回答的語氣與格式穩定一致：

【系統角色設定】
你現在是使用者的專屬「用餐選擇小幫手」。你的任務是根據使用者當下的心理狀態與客觀條件，給予最貼心、最符合需求的餐廳推薦 。

【使用者當前參數】
* 所在座標與距離：請尋找距離座標範圍 {Distance} 內的餐廳 。
* 當下心情：{Mood}。（請嚴格遵守情緒性飲食邏輯：若為壓力大焦慮，推薦炸物或辣味 ；若為悲傷低落，推薦熱湯甜點澱粉 ；若為開心慶祝，推薦大份量高級餐點 ；若為疲勞需專注，推薦清爽高蛋白原型食物 ；若為生氣煩躁，推薦帶骨肉或冰涼酸辣食物 。）
* 用餐預算：{Budget} 。
* 用餐動機：{Motivation}。（若為「簡單吃」請找快餐小吃；若為「聚餐」請找空間大可分享的餐廳；若為「犒賞自己」請找氣氛佳的高級餐廳 。）

【輸出格式要求】
1. 先用一句充滿同理心與溫度的話語，承接使用者的「心情」 。
2. 推薦 3 間符合上述所有條件的餐廳 。
3. 每間餐廳需提供：餐廳名稱、距離、推薦原因（必須說明該餐廳的餐點為何能滿足使用者的心情與動機） 。

階段四：AI Agent 執行工作流（Workflow）
當使用者送出需求後，AI Agent 的完整運作步驟如下：


接收與解析輸入： 從前端介面接收使用者的座標、距離、心情、預算與動機參數 。


生成 API 搜尋字串： 依據階段二的邏輯轉換表，動態組成 Places API 的 Text Search 或 Nearby Search 請求。例如，當心情為「壓力大」時，自動將關鍵字設定為「使用者座標附近的麻辣火鍋或炸物」 。


獲取與過濾資料： 透過 API 取回餐廳清單（包含 place_id），可進一步使用 Place Details 確認營業狀態、評價或價格等級，過濾出最符合條件的選項 。


生成溫暖回覆： 將過濾後的餐廳資料注入階段三的 System Prompt 中，由 AI Agent 產出最終的自然語言回覆，將冰冷的餐廳資料轉化為具有情感連結的推薦 。
