# 用餐選擇小幫手 - 交接文件

## 專案目標 (Project Goal)
打造一個手機優先的網頁 App。使用者一開始連線先看到全螢幕首頁（大圖＋定位／開始推薦按鈕），按下開始後依序進入會員資料設定（僅首次）→ 心情/動機/價位/距離查詢 → 透過 Google Places API 取得餐廳資料，並由 LLM 生成具同理心的自然語言推薦（3 間餐廳），支援未選中時的次輪放寬推薦。

## 已完成進度 (Completed)

### 程式碼結構
```
src/
  lib/
    types.ts        # 所有共用型別（Profile、RecommendRequestBody、RestaurantCard...）

    places.ts        # 呼叫 Google Places Text Search、距離計算、導航連結組字串
    recommend.ts      # 核心業務邏輯：首輪精準搜尋、保底救援、共用的完整降級放寬引擎（換一批／放寬搜尋共用）
    storage.ts        # localStorage 讀寫（profile.json / history.json 概念對應）
  app/
    page.tsx                  # 主頁面，管理整個流程的狀態機（home → profile-setup → returning-prompt → query → results）
    components/
      HomePage.tsx             # 首頁（連線後第一個畫面）：全螢幕大圖＋定位／開始推薦按鈕
      ProfileForm.tsx          # 會員資料設定表單
      QueryForm.tsx            # 心情/動機/價位/距離查詢表單（定位已在首頁取得，這裡只接收）
      ReturningUserPrompt.tsx  # 回頭客詢問（避開前次餐廳／滿意度回饋）
      ResultsView.tsx          # 推薦結果顯示（首輪+次輪同頁捲動、餐廳卡片、保底提示）
    api/
      recommend/route.ts       # POST，首輪（round=1）與換一批（round=2）的後端進入點
      recommend/widen/route.ts # POST，「放寬搜尋」補足動作的後端進入點（見下方說明）
      photo/route.ts           # GET，餐廳照片代理（見下方說明）
```

### 已實作並端對端測試通過的功能（已用 OpenAI `gpt-4o-mini` 版本實測，`/api/recommend` 全流程正常）
- 首頁（`HomePage.tsx`）：使用者一連線就先看到，取得定位＋按「開始推薦」後才進入會員資料/查詢流程。
- 會員資料設定（年齡、性別、健康方向、飲食限制、辣度接受度）。
- 心情（6 類）＋動機（5 類）＋價位＋距離的查詢表單（定位已在首頁取得）。
- 後端依 Tier 表組基礎關鍵字 → 呼叫 LLM 依會員資料微調成最終搜尋字串 → 呼叫 Places API 搜尋 → 硬性條件過濾（素食欄位、海鮮/堅果過敏關鍵字避開）→ 價位符合優先＋距離排序。
  - 效能優化：搜尋字串只在 Tier1 呼叫一次 LLM，Tier2/Tier3 用字典字串組合不呼叫 LLM；首輪算出的搜尋字串（`tierQueries`）會原封不動回傳給前端，「換一批」「放寬搜尋」都由前端把它原樣帶回來重複使用，不重新呼叫 LLM（因為這組字串跟距離/價位無關）。
- **首輪／換一批／放寬搜尋** 三段式機制（`recommend.ts`）：
  - **首輪（精準）**：只查 Tier1，不夠3家才查 Tier2（若該心情沒有 Tier2 則略過），**不**做距離/價位放寬，允許顯示 0～3 家（不強行湊數）。
  - **保底救援**：只有 Tier1+Tier2 完全查無結果（0筆）時才觸發，依序改查 Tier3（原距離）→ 還是0筆則距離+1km 再查一次 Tier3（價位不動），確保下限盡量拉到至少1筆，避免使用者一開始就看到空白畫面。回應會帶一個 `rescued` 旗標告知前端有沒有觸發保底。
  - **換一批**（使用者按「都不喜歡，幫我換一批」）／**放寬搜尋**（使用者在保底提示按「放寬搜尋」）：共用同一套「完整降級放寬引擎」（`runFullRelaxCascade`）：Tier1→2→3 依序查到湊滿需要的筆數，還不夠才放寬距離+1km、價位+1級再查一次。差別只在需要的筆數與排除清單——「換一批」要湊滿3家全新的（放在獨立的「次輪推薦」區塊）；「放寬搜尋」只補首輪目前顯示數與3家的差額，**併入首輪同一份清單，不取代已顯示的餐廳**。
  - 首輪選擇少於3家時，畫面顯示保底提示：「放寬搜尋」／「重新推薦」（回到心情/動機輸入畫面）兩個按鈕。文案依「真實0筆」與「真實1、2筆」分開撰寫，避免開場白與提示文字重複講同一句「要不要放寬」。
- 首輪未選中可另外觸發次輪：排除首輪已顯示的餐廳，湊一批全新的3家，**新版 UI 為單頁往下捲動**（次輪結果接在首輪下方，載入完成自動平滑捲動到次輪區塊，取代原本的分頁切換設計，對手機使用者更直覺）。
- 選中餐廳後提供 Google 導航連結（免另外呼叫 API，純組字串）。
- 回頭使用者會被詢問「避開前次餐廳」或「回饋前次滿意度」，寫入 `localStorage`。
- LLM 回覆為**結構化輸出**（`opening` 開場同理心語句＋每間餐廳各自的 `reason`），不再是一整段文字：
  - 開場語句獨立用斜體襯線字呈現，跟餐廳卡片視覺區隔。
  - 各餐廳的推薦理由直接放進該餐廳的卡片內。
- 餐廳卡片顯示：商家照片（見下）、名稱、距離、主要餐點類別（`primaryTypeDisplayName`）、價位（＄符號數量）、地址、推薦理由、選擇/導航按鈕。
- **商家照片代理路由**（`/api/photo`）：Google 照片網址需要帶 API 金鑰才能存取，不能直接讓前端 `<img>` 打 Google 網址（會外洩金鑰）。做法是後端自己多開一條路由，前端只拿到自家網址（如 `/api/photo?name=...`），實際跟 Google 要圖、帶金鑰的動作在後端完成，並驗證 `name` 參數格式避免被亂打參數濫用。
- 已修正深色模式造成「白字白底看不到」的 CSS bug（`globals.css` 移除跟隨系統深色模式的邏輯，加上 `color-scheme: light` 讓瀏覽器原生表單元件也強制淺色呈現）。
- **距離硬性上限**：發現 Google Places `locationBias` 只是軟性偏好、有時會回傳遠超使用者設定距離的結果（實測案例：設定範圍內卻推薦到 14.3 公里外的店），因此在 `recommend.ts` 加入硬性過濾。上限依交通工具的直覺分級（步行/機車/開車誤差容忍度不同），採三段倍數：輸入距離 ≤2km 時上限為輸入值 ×1.2、2–5km 時 ×1.5、>5km 時 ×2。首輪與次輪放寬都套用同一個上限（以原始輸入距離為準，不隨次輪放寬而跟著放大——這是刻意設計，距離上限代表使用者選擇距離時隱含的交通工具可達範圍，不會因為「放寬動作」或「使用者主動選擇進入次輪」而跟著放大，這兩者是「預期管理」跟「物理可達性」兩個不同的問題）。已用 API 直接測試驗證（含次輪放寬情境下候選不足時，正確地只回傳符合上限的餐廳，不會為了湊滿 3 間硬塞超遠結果）。
- **`rankPreference: "DISTANCE"` 與距離滑桿下限調整為 1km**：用真實 API 呼叫（非 mock）測試發現，locationBias 對現行多關鍵字組合查詢字串（tier1 LLM 輸出、tier2/tier3 拼接格式）幾乎完全失效——8組測試（都市/郊區×2距離×2字串）中，落在使用者原始設定距離內的結果恆為0，超過放寬半徑仍被回傳的比例高達17~21/20~21筆。查 Google 官方文件確認 Places Text Search（New）支援 `rankPreference: "DISTANCE"` 排序參數，實測後在郊區2km情境有顯著改善（從「2筆合法+19筆超出」變成「20筆全部落在放寬半徑內」），已加入 `src/lib/places.ts` 的 `searchPlacesText`。同時發現極短距離（0.5km）不論查詢字串或排序參數都無法改善——這是 Places API 對極小圓形 bias 的結構性限制，判斷與其讓使用者以為「500m內應該有很多選擇」但系統做不到，不如在選項層級就不提供，因此把 `QueryForm.tsx` 距離滑桿下限從 0.5km 改成 1km。已用 Playwright 跑 5 輪真實端對端測試（都市/郊區各數輪，含點擊「放寬搜尋」），確認：先前真實壞掉的案例（郊區/2km/疲勞+簡單吃，原本應為真實0筆）改善成找到2家、推薦理由語意相關性沒有變差、1km 下限確實生效（含瀏覽器原生 range input clamp 行為）、保底救援畫面在真的很少店的組合下依然正確觸發、5輪測試 server log 全部 200、console 全部 0 錯誤。原始 API 量化數據記錄在本機 `本機測試記錄.md`（不進 git，僅供內部分析）。
- **營業時間硬性篩選**（2026-07-16）：晚間使用時曾推薦到已打烊的餐廳。`src/lib/places.ts` 的 `searchPlacesText()` request body 加上 `openNow: true`，由 Google 伺服器端直接排除未營業候選，round1/換一批/放寬搜尋共用同一個函式故自動套用，無需另外改推薦邏輯。`tsc --noEmit`、`eslint` 皆通過；邏輯先在 `food-recommendation` 專案用真實 API 呼叫驗證過（同查詢比對，不篩選時20筆裡有5筆實際已打烊，加篩選後0筆漏網），兩專案 request body 完全一致，故本專案未另外重跑一次即時 API 驗證。已知取捨：Google 對沒登錄營業時間資料的店家也會一併排除，非單純排除「查到打烊」的，在地小吃店較容易受影響，候選池可能變小、更容易觸發保底救援。此篩選不會被放寬搜尋鬆綁，跟距離/價位放寬不同，行為與素食/過敏硬性篩選一致。

### 使用體驗優化
- **距離上限分級**：見上方「距離硬性上限」，把原本單一 2 倍的粗略上限，依距離區間拆成三段更貼近實際交通工具誤差容忍度的倍數。

### 心情（6 類）與動機（5 類）關鍵字對照表
**心情**（Tier1→Tier2→Tier3 逐層放寬，候選不足 3 家才往下一層）：

| 心情 | Tier 1 | Tier 2 | Tier 3（保底） |
|---|---|---|---|
